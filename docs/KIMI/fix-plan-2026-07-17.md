| 阶段 | 目标 | 预计工作量 | 项数 |

## P0 — 立刻修（安全 + 死功能 + 数据污染）

### P0-1. 修复 body 上限矛盾（评审 R1）

- 全局保持 `express.json({limit:'1mb'})`；
- 对 `/api/v1/dj/asr`、`/api/v1/dj/analyze-image` 单独挂 `express.json({limit:'25mb'})`（在该路由挂载之前注册，路径级 body-parser 先于全局生效）；
- `middleware/error.ts` 识别 `entity.too.large` 错误类型，返回 413 而不是 500；
- 验证：手动发一个 >1MB 的 base64 请求到 `/api/v1/dj/asr`，确认不再 413/500 误报。

### P0-2. 鉴权改为 fail-closed（评审 R2）

- `middleware/auth.ts`：只有显式 `NODE_ENV=development` 才允许无 `API_KEY` 放行；
- 启动时校验：非 development 且无 key → 直接拒绝启动（复用/扩展 `assertSecretConfigured`）；
- `sessionToken.ts` 的 `DEV_FALLBACK_SECRET` 同样加 development 限定，非开发环境直接抛错；
- 验证：不设 `NODE_ENV` 和 `API_KEY` 启动，应拒绝启动；设 `NODE_ENV=development` 启动正常。

### P0-3. 修复收藏上报 action 反向（评审 F1）

- `components/KimiCard.tsx:130-148`：`toggleLike()` 后改用 `useRadioStore.getState().likedSongIds.includes(id)` 读取切换后的新值（或干脆取反旧值），再进入 debounce 上报；
- 顺手清理后端 feedback 表中已写反的历史数据（或标记作废，视数据量决定）；
- 补一个单测断言上报的 action 与新状态一致；
- 验证：点收藏 → 后端日志 `Feedback: like`；取消 → `Feedback: unlike`。

### P0-4. aiLimiter 改为按路由挂载（评审 B1）

- 从 `app.use('/api/v1/radio', aiLimiter, radioRoutes)` 拆下；
- 只挂在具体 POST 路由：`create`、`next`、`chat`、`/dj/tts`、`/dj/intro`、`/dj/asr`、`/dj/analyze-image`；
- GET（models/songs/queue）与 feedback 不挂 aiLimiter，feedback 保留自己的 30/分钟 limiter；
- 验证：连续 15 次 GET `/api/v1/radio/:id/queue` 不 429；feedback 30 次/分钟内不 429。

### P0-5. 端口口径全仓统一（评审 B7 / C1）

- `backend/.env.example`：`PORT=8000` → `PORT=8001`；
- `backend/src/config.ts:15` CORS 默认白名单补 `http://localhost:3000`；
- `start.sh` / `start.bat` / `start.ps1` 三处打印的前端地址 3001 → 3000（或统一改为显式 `next dev -p 3001`，二选一，推荐前者改动最小）；
- 同步修正 `D:\Coder\AGENTS.md` 中"mimo-radio 前端 3001"的描述。

### P1-2. 前端四处竞态/泄漏修复（评审 F2 / F3 / F4 / F5）

### P1-3. helmet 配置单一来源（评审 B5）

### P1-4. 缓存与数据清理（评审 B6 / B7）

## P2 — 顺手做（仓库卫生 + 架构决断）

### P2-1. UPnP 二选一决断（评审 B4）

**推荐：先下线。** 删除 `routes/upnp.ts`、`services/upnp.ts`、`upnp-device-client` 依赖及入口注册。理由：当前 play() 是 stub 却返回 `ok:true`，"谎报成功"的接口比没有接口更糟；且 SSRF 私网拦截与 UPnP 天然冲突，实现前必须先补 `ssrfGuard` 的 DNS 解析校验（`dns.lookup` 验证真实 IP + IPv6 方括号形态），成本不低。要用时按"ssrfGuard 补 DNS 校验 → LAN 白名单 → 实现 play"的顺序加回来。

### P2-2. 仓库卫生（评审 C2 / C3 / C6）

- `backend/tsconfig.json` `exclude` 加 `src/**/*.test.ts`（或拆 `tsconfig.build.json`），确认 `dist/` 不再含测试；
- `.gitignore` 补：`*.tar.gz`、`*.pid`、`backend/static/audio/`、`frontend/public/sw.js`、`frontend/public/workbox-*.js`；
- `git rm --cached` 清理已跟踪的 `HANDOVER.tar.gz`、`backup_src_*.tar.gz`、`frontend/frontend.pid`、TTS mp3、`sw.js`、`workbox-*.js`（注意：git 历史中的体积仍在，如需彻底清除要 filter-repo，可暂缓）；
- 删除 `backend/nul`、`frontend/nul`；清理 `logs/` 下调试残留；
- 删除死代码：`TtsEngineSwitcher.tsx`、`MarkdownText.tsx`、`neteaseCookie` 死配置、`db/index.ts` 的 `getSongs/setSongs`、`@types/ws` 残留依赖；
- 根 package.json 补 `test`、`lint` 聚合脚本。

### P2-3. 测试补齐 + 文档更新（评审 C4 / C5）

- 给 `services/planner.ts` 补测试（重点：`generateDailyPlan` 缓存命中/未命中、并发去重——顺手把 `planner.ts:62` 的并发重复调 AI 问题一起修）；
