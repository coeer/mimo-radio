### ❌ 这次的问题

#### 问题 1:F8(进度条漂移)是误报 ⚠️

你写"PlayerBar 用 setInterval 本地计时,无校准"。但实际代码 `PlayerBar.tsx:17-19` **已经有校准**:
```ts
useEffect(() => { setLocalTime(currentTime) }, [currentTime])  // ← currentTime 变就重置
```
你的"改后代码"和现有代码几乎一样,只是多了 prevTimeRef 的 seek 检测。**现有逻辑已经能校准 currentTime 变化**(包括 seek,因为 seek 会改 currentTime 触发这个 effect)。唯一边缘场景是"播放中 currentTime 持续更新 + 本地 interval 持续 +1 导致短暂叠加",但 setLocalTime 重置会覆盖,不构成"漂移"。

**判定**:**误报**。F8 删掉,或降级为"⚠️ 边缘优化(1s 内的视觉抖动),非 bug"。

#### 问题 2:F1 根因解释不准确(改法对,解释含糊)

你说"`useRadioStore((s) => s.isLiked)` 选择了函数引用,Zustand Object.is 比较函数引用不变 → 不 re-render"。**这个机制描述含糊**。

精确根因:`isLiked` 是 store 的一个 action 函数(`radioStore.ts:146` `isLiked: (id) => get().likedSongIds.includes(id)`)。组件订阅 `s.isLiked`(函数引用),函数引用永远不变,所以 Zustand 不会因为 `likedSongIds` 数组变化而通知这个组件 re-render。**问题不是"选了函数引用"本身,而是组件订阅了一个不变化的派生函数,却期望它反映变化的派生数据**。

改法(改订阅 `likedSongIds.includes(id)`)是对的——因为这样订阅的是数组本身,数组变就 re-render。**结论正确,但要把根因讲清楚**:不是"Object.is 比较函数引用",是"订阅了错误的订阅源"。

#### 问题 3:F3 REPLAY 改法有点取巧

你给 REPLAY 加后缀"现在是《歌名》"。这解决"歌名滞后",但**有点 patch 味**——DJ 说一段旧的串词,然后突然冒出"现在是《另一首》",听感突兀。

更好的方向(供你思考,不强制):REPLAY 重播时,如果当前歌和消息里的歌不一致,**禁用 REPLAY 按钮**(或灰显),而不是播了再补一句。因为重播一段"关于别的歌的解说"本身就是错位的,补后缀是承认错位。

## 修复方案修正(给执行者的最终版)

### 删除 F8(误报)
PlayerBar.tsx:17-19 已有 `setLocalTime(currentTime)` 校准。删掉 F8 任务。

### F1 根因解释修正
不是"Object.is 比较函数引用",是"**订阅了派生函数而非派生数据**"。改法不变(订阅 likedSongIds)。

### F2 改法优化(用 ref,更直接)
你给的改法 A(退出时读 current data-theme 判断)逻辑能跑,但有个隐藏问题:如果用户在全屏期间通过某种方式(理论上 UI 藏了)切了主题,你的逻辑会"保留用户切的主题",但这不是全屏该负责的。**更直接的正解是 useRef**(我之前的方案):

```tsx
const prevThemeRef = useRef('dark')
useEffect(() => {
  const root = document.documentElement
  prevThemeRef.current = root.getAttribute('data-theme') || 'dark'
  root.setAttribute('data-theme', 'light')
  return () => { root.setAttribute('data-theme', prevThemeRef.current) }
}, [])
```
ref 跨渲染保持,cleanup 读 ref.current 拿 mount 时记录的真实值。**简单、直接、无歧义**。你的改法 A 也能工作,但 ref 更不容易出边界问题。

### F3 改法待定(建议禁用按钮而非补后缀)
两个选择,你定:
- **选项 1(你的方案)**:REPLAY 追加"现在是《歌名》"——能工作,但听感突兀
- **选项 2(推荐)**:REPLAY 按钮在"消息关联的歌 ≠ 当前歌"时禁用/灰显——避免错位重播

倾向选项 2,但如果你判断"用户就是想重听那段解说",选项 1 也接受。
